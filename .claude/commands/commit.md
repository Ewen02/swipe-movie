Create an atomic conventional commit for the Swipe Movie project with pre-verification.

If `$ARGUMENTS` is provided, use it as a hint for the commit description.

---

## Phase 1: Pre-verification

Run these checks **before** committing. If any fails, report errors and do NOT commit.

1. Run `git status` to see staged and unstaged changes
   - If **nothing is staged**: list the changed files and ask the user what to stage. Do not proceed until files are staged.
   - If files are staged: continue

2. Run `pnpm typecheck` to verify TypeScript compiles
   - If it fails: show the errors, suggest fixes, stop

3. Run lint on staged `.ts`/`.tsx` files:
   ```
   git diff --cached --name-only --diff-filter=ACMR -- '*.ts' '*.tsx' | xargs -r pnpm eslint --fix
   ```
   - If ESLint reports unfixable errors: show them, stop
   - If ESLint auto-fixed files: re-stage the fixed files with `git add`

If all checks pass, proceed to Phase 2.

---

## Phase 2: Analyze & classify

1. Run in parallel:
   - `git diff --cached --stat` — summary of changed files
   - `git diff --cached` — full diff of staged changes
   - `git log --oneline -10` — recent commits for style reference

2. Determine the **scope** from file paths:
   | Path prefix | Scope |
   |---|---|
   | `apps/web/` | `web` |
   | `apps/api/` | `api` |
   | `packages/ui/` | `ui` |
   | `packages/types/` | `types` |
   | `packages/database/` | `database` |
   | `packages/email/` | `email` |
   | `packages/subscription/` | `subscription` |
   | `docker/` | `docker` |
   | `.github/` | `ci` |
   | Root files (`.claude/`, `package.json`, `pnpm-lock.yaml`, `turbo.json`, etc.) | omit scope |

3. Determine the **type** from the nature of changes:
   - `feat` — new functionality, new files with business logic
   - `fix` — bug fix, error correction
   - `refactor` — restructuring without behavior change
   - `perf` — performance improvement
   - `test` — adding/modifying tests
   - `docs` — documentation only
   - `style` — formatting, whitespace, no logic change
   - `chore` — dependencies, config, tooling, CI

4. Write a **description**: concise, imperative, lowercase first letter, no trailing period. Focus on *why* not *what*.

5. **Atomic check**: If staged changes span **multiple scopes**, suggest splitting into separate commits — one per scope. Ask the user if they want to:
   - Commit all together (pick the primary scope)
   - Split into atomic commits (you will unstage, re-stage per scope, and commit each)

---

## Phase 3: Commit

1. Present the proposed commit message to the user:
   ```
   type(scope): description
   ```

2. Ask for confirmation or edits

3. Create the commit using HEREDOC:
   ```bash
   git commit -m "$(cat <<'EOF'
   type(scope): description
   EOF
   )"
   ```

4. Run `git status` to confirm the commit succeeded

---

## Rules (STRICT)

- **Format**: `type(scope): description` — always
- **Lowercase** first letter after colon
- **No trailing period**
- **NEVER** mention Claude, Claude Code, AI, assistant, copilot, or "generated by" in the commit message
- **NEVER** add a `Co-Authored-By` line or any AI attribution trailer
- **NEVER** use `git add -A` or `git add .` — always add specific files
- **NEVER** use `--no-verify` — let hooks run
- **NEVER** amend unless the user explicitly asks
- If `$ARGUMENTS` is provided and looks like `type description` (e.g., `fix login redirect`), use the first word as type hint and the rest as description hint. Still verify the type is valid and the scope is correct.
